# v0.2.0 plan: Dandelion++ adversarial experiment suite

**Status:** Draft (2026-05-21)
**Target:** First feature release after v0.1.0
**Theme:** simulate the centralized DNS-blocklist trust mechanism and the
adversarial network conditions it's designed to defend against.

## 1. Goal

Make it possible to run end-to-end Dandelion++ adversarial experiments in
monerosim:

- Publish a sim-time-scheduled ban list (mimicking `blocklist.moneropulse.*`)
- Push it to subscribed nodes via `set_bans` RPC (no monerod patches)
- Observe stem-phase transactions from adversary-controlled "eavesdropper"
  nodes
- Compute deanonymization rates conditioned on ban-list state, attacker
  connectivity, and propagation cadence

The release ships the simulation primitive *and* the analysis tooling, so
v0.2.0 is the first version of monerosim that can produce paper-grade D++
results out of the box.

## 2. Non-goals

- Modifying monerod or shadowformonero. The whole approach is RPC-driven.
- Bypassing the `--enable-dns-blocklist` mainnet gate. We simulate the
  *effect* (banned IPs in `block_subnet`/`block_host`), not the literal
  DNS-polling code path inside monerod.
- Multi-publisher / competing-trust-root scenarios. Deferred to v0.3.0.
- Anything that touches the v0.1.0 config schema in a breaking way. All
  new fields are optional with off-by-default semantics.

## 3. Background

monerod's `--enable-dns-blocklist` polls TXT records on
`blocklist.moneropulse.{se,org,net,co,fr,de,ch}` every 7000 s, parses
semicolon-separated IP/CIDR entries, and calls `block_subnet()` /
`block_host()` per entry (see `net_node.inl:2147`). That mechanism is
hard-gated to `MAINNET` (`net_node.inl:2151`), but monerosim runs
regtest so the code path is dead.

`generateblocks` (the synthetic-mining RPC) is gated to `FAKECHAIN`
(`core_rpc_server.cpp:2347`), so we can't move off regtest without losing
mining. The two features are mutually exclusive by nettype.

Workaround: drive the same `block_subnet`/`block_host` operations via the
`set_bans` RPC (mainnet-or-regtest, available for years), from a central
dispatcher that polls the in-sim DNS server. Network-level effect is
byte-identical to native `--enable-dns-blocklist`.

## 4. Architecture

                   ┌─────────────────────────┐
                   │  ban_schedule.yaml      │
                   │  (sim-time scheduled    │
                   │   publish/expire ops)   │

                                │ reads
                                ▼
                   ┌─────────────────────────┐
                   │  publisher agent        │

                   └────────────┬────────────┘
                                │ writes
                                ▼
                   ┌─────────────────────────┐
                   │  current_banlist.json   │
                   └────────────┬────────────┘
                                │ reads

                   │  DNS server             │

                   ┌─────────────────────────┐

## 5. Components

### 5.1 Ban schedule format

A new YAML file referenced from the scenario:

```yaml
# ban_schedule.yaml
schedule:
  - sim_time: 1h
    publish:
      - ip: 11.0.0.42
        ttl: 24h
      - subnet: 11.0.0.0/24
        ttl: 6h
  - sim_time: 3h
    publish: []           # empty = expire everything
  - sim_time: 4h
    publish:
      - ip: 11.0.0.99
        ttl: 12h
```

Semantics:
- Each `sim_time` event *replaces* the published set (snapshot semantics,
  matching how monerod consumes DNS TXT records).
- `ttl` per entry maps to the `seconds` argument of `set_bans`.
- Durations accept the same string format as the rest of the scenario
  config (`6h`, `300s`, etc.).

### 5.2 DNS server extension (`agents/dns_server.py`)

Add a new TXT-record zone for the seven `blocklist.moneropulse.*` domains
listed in `net_node.inl:2153-2161`. Records are loaded dynamically from
`current_banlist.json` (rewritten by the publisher) on each query, so
updates take effect immediately without DNS server restart.

Output format matches monerod's parser:
- Multiple TXT records OK
- Each record is a `;`-separated list of IPs or CIDR subnets
- Reuse the existing TXT-record machinery (already present for the
  `checkpoints.moneropulse.*` stub)

The new zone only activates when `dns_blocklist.enabled = true` in the
scenario.

### 5.3 Publisher agent (`agents/dns_blocklist_publisher.py`, new)

Single process per scenario. Lifecycle:
1. On startup: parse `ban_schedule.yaml`, write the initial empty state.
2. Sleep until the next scheduled `sim_time` event.
3. Wake, rewrite `current_banlist.json` with the new published set.
4. Loop.

No RPC, no network. Purely a sim-time-driven file rewriter. Logs every
event for the post-run record.

### 5.4 Dispatcher agent (`agents/dns_blocklist_dispatcher.py`, new)

Single process per scenario. Simulates the aggregate behavior of every
subscribed node's `update_dns_blocklist()` loop. Lifecycle:

```python
while True:
    # 1. Poll DNS for the current ban list.
    bans = resolve_blocklist_txt(SEED_DOMAINS)   # returns list[IpOrSubnet]

    # 2. For each subscriber, dispatch at t + uniform(0, poll_interval)
    #    to simulate independent polling jitter.
    deadline = now + poll_interval
    for subscriber in subscribed_nodes:
        jitter = random.uniform(0, poll_interval)
        schedule_at(now + jitter, lambda s=subscriber: push_bans(s, bans))

    # 3. Sleep until the next poll boundary.
    sleep_until(deadline)
```

Per-subscriber RPC errors don't kill the dispatcher — that node just
misses the cycle and gets the update on the next poll. This matches real
operator behavior (transient DNS failures, daemon restarts, etc.).

The subscriber list is built once at startup by reading the agent
registry and filtering by `dns_blocklist_subscribed` and/or the
scenario-level `subscriber_fraction` / `subscriber_agent_types` config.

### 5.5 RPC client extension (`agents/monero_rpc.py`)

Add (or surface, if already present):

```python
def set_bans(self, bans: list[BanEntry]) -> None:
    """Push a complete ban list to the daemon.

    Each BanEntry is {host: str, ban: bool, seconds: int}. The call is
    additive — entries not in `bans` are not unbanned. To clear the list,
    call get_bans() first and pass each entry with ban=False.
    """

def get_bans(self) -> list[BanEntry]:
    """Read current ban state from the daemon."""
```

Investigation required in Phase 1: confirm whether these wrappers already
exist; if so, just add type hints and a smoke test.

### 5.6 Adversarial observer agent (`agents/stem_observer.py`, new)

A new agent type for "passive eavesdropper" nodes. They participate in
P2P as ordinary relays — accepting connections, propagating blocks — but
log every incoming transaction with:
- `sim_time`
- `tx_hash`
- `source_peer_ip`
- `was_stem_phase` (derived from D++ headers if available, or inferred
  from propagation pattern)
- `local_peer_count`

Output goes to `stem_observer.log` in the agent's data directory,
JSONL-formatted for easy ingestion.

Observer nodes are configured as a new agent type in the scenario:

```yaml
agents:
  - type: stem_observer
    name: observer-{001..050}
```

They count toward `total_nodes` for resource calculations but are
otherwise plain monerod daemons with extra RPC subscriptions for tx
notifications.

### 5.7 tx-analyzer extension

Add a new subcommand to the existing tx-analyzer (`scripts/`):

```bash
python3 scripts/tx_analyzer.py deanon-rate \
    --observers archive/daemon_logs/observer-*/stem_observer.log \
    --ground-truth archive/agent_activity.jsonl \
    --output dandelion_report.txt
```

Output:
- Per-tx classification: correctly attributed / incorrectly attributed /
  observed-but-unattributed / unobserved
- Aggregate deanonymization rate
- Breakdown by ban-list state (before first ban / during ban / after expiry)
- Breakdown by attacker connectivity (fraction of honest nodes the
  attacker peers with)
- Optional: confidence intervals via bootstrap resampling

Existing tx-analyzer stem-path reconstruction logic is reused; this new
subcommand layers analytics on top of it.

## 6. Scenario schema additions

All new, all optional, all default to off:

```yaml
# Top-level
dns_blocklist:
  enabled: true                         # default false
  schedule_file: ban_schedule.yaml
  poll_interval: 60s                    # subscriber cadence
  subscriber_fraction: 1.0              # 0.0-1.0
  subscriber_agent_types: [user, relay] # which kinds count
  dispatcher_seed: 12345                # for reproducible jitter

# Per-agent (optional override)
agents:
  - name: relay-042
    type: relay
    daemon_settings:
      dns_blocklist_subscribed: true    # explicit; wins over fraction
```

Adversarial-observer schema reuses the existing agent type machinery:

```yaml
agents:
  - type: stem_observer
    name: observer-{001..050}
    daemon_settings:
      dns_blocklist_subscribed: false   # observers usually don't subscribe
```

## 8. Test plan

### 8.1 Unit / integration

- `test_dns_blocklist_publisher.py` — schedule advances correctly, file
  rewrites match expected snapshots
- `test_dns_blocklist_dispatcher.py` — jitter distribution is uniform
  across the poll interval; per-node `set_bans` calls land within
  bounds; RPC errors don't poison the loop
- `test_dns_server.py` — TXT zone responds correctly, falls back when
  banlist file missing, format matches monerod parser expectations
- `test_monero_rpc.py` — set_bans/get_bans roundtrip on a real regtest
  monerod (existing test infrastructure)

### 8.2 Smoke scenario (`test_configs/dns_blocklist_smoke.yaml`)

- 5 honest miners
- 10 user agents
- 5 relays (3 subscribed, 2 not — to verify per-agent override)
- 1 publisher, 1 dispatcher, 1 DNS server
- 1 ban event at sim_time=10m
- 30m total runtime
- **Pass condition**: subscribed daemons have the banned IP in `get_bans`
  output after the dispatcher's first poll cycle; unsubscribed daemons
  do not.

### 8.3 Research scenario (`test_configs/dandelion_adversarial.yaml`)

- 5 miners, 100 users, 200 relays, 30 observers
- 1 publisher with realistic ban schedule (bans at 1h, 3h; expires at 6h)
- 8h runtime
- **Pass condition**: smoke level only — verifies the scenario runs to
  completion and tx-analyzer can produce a report. The actual research
  numbers are for the user to interpret, not for CI.

## 9. Non-breaking guarantees

- Every new scenario field is optional and defaults to "off"
- `dns_blocklist:` section absence → no publisher, no dispatcher, no DNS
  TXT zone, no new processes
- Existing scenarios produce byte-identical orchestrator output
  (verified by `test_configs/quickstart.yaml` golden-file test)
- No new monerod CLI flags
- No new shadowformonero patches
- No changes to `daemon_defaults` semantics

## 10. Future work (v0.3.0 and beyond)

- **Multiple competing publishers** — for ban-list capture / split-trust
  experiments
- **Realistic propagation lag** — model the time between an operator's
  published-on-DNS update and propagation to all monerod instances of the
  world, beyond just the polling interval
- **Adversarial publisher mode** — publisher with a malicious schedule
  that targets specific honest nodes
- **Native polling via patched monerod** — if it ever becomes worth the
  fork-maintenance cost, swap the dispatcher for a one-line monerod
  patch that removes the MAINNET gate

## 11. Open questions

- **Subscriber heterogeneity model**: should `subscriber_fraction` apply
  uniformly across all `subscriber_agent_types`, or per-type? Default
  proposal: uniform; per-type override possible via explicit
  `dns_blocklist_subscribed: bool` per agent.
- **Dispatcher RPC parallelism**: serial vs. parallel `set_bans` calls.
  Serial is simpler and probably fast enough for ~1000 subscribers per
  poll cycle. Investigate during Phase 4 if it becomes a bottleneck.
- **Observer tx-source detection**: how do we distinguish stem-phase from
  fluff-phase transactions at the observer? monerod logs the `dandelion`
  flag in some cases; may need to extend logging or infer from
  propagation pattern. Investigate during Phase 6.
- **Agent registry IP discovery**: confirm the dispatcher can read each
  subscriber's IP and RPC port from the existing registry without new
  infrastructure. Investigate during Phase 4.

## 12. Version policy

This is a **feature release**: new agent types, new config schema fields,
new analysis commands. Per the v0.1.0 banner, that's a `0.x.0` minor
bump → **v0.2.0**. All additions are config-compatible with v0.1.0
scenarios (which simply don't use the new fields), so a v0.1.x patch
release could in principle include this, but the scope is large enough
that minor-bump signaling is more honest about the change.
